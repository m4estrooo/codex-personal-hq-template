# AGENTS.md

Правила этого репозитория обязательны для Codex и всех субагентов, работающих в текущем проекте.

## Назначение репозитория

Это шаблон личного штаба: задачи, handoff между чатами и устройствами, проекты, исследования, правила Codex и будущая автоматизация.

## Язык и стиль

- Пиши по-русски.
- Если используешь английский термин, при первом упоминании дай русский смысл в скобках.
- Не выдумывай факты, ссылки, цены, статусы объявлений, наличие товаров, характеристики сервисов или правила платформ.
- Если факт может устареть, проверь его через интернет, браузер, официальный источник или явно напиши, что проверка не выполнена.

## Планирование

- Для любой задачи длиннее 1-2 действий создай и обновляй видимый план Codex.
- Если план-трекер недоступен, веди Markdown-чеклист в активном файле задачи.
- План должен показывать реальные этапы: контекст, реализация, проверка, handoff/документы.

## Задачи и handoff

- Любой новый зародыш идеи, задачи или проекта сразу фиксируй в `tasks/inbox.md`, даже если деталей пока мало. Не оставляй идеи только в чате.
- Нормальный жизненный цикл задачи: `tasks/inbox.md` -> `tasks/active/<task-id>.md` -> `tasks/archive/<task-id>.md`.
- `tasks/inbox.md` - единый список входящих и неактивных задач: тут лежат и сырые идеи без достаточной проработки, и понятные задачи, которые пока не взяты в работу.
- `tasks/backlog.md` не используется как обязательный этап. Не переноси задачи в backlog только потому, что они стали понятнее; оставляй их в inbox до выбора в active.
- Когда идея принята в активную работу и получает файл в `tasks/active/`, удаляй ее из `tasks/inbox.md`, чтобы одна задача не висела сразу в двух состояниях.
- Если пользователь явно называет задачу текущей/основной, можно переводить ее из inbox сразу в active, но это нужно зафиксировать в task/handoff.
- Каждая активная задача получает стабильный `task-id`.
- Файл задачи: `tasks/active/<task-id>.md`.
- Файл передачи контекста: `handoff/active/<task-id>.md`.
- `handoff/current.md` - рабочая доска текущих контекстов, а не единственный указатель. В нем может быть несколько активных фокусов одновременно: задачи штаба, дочерние репозитории, связанные task/handoff-файлы, последний статус и следующий шаг по каждому.
- Если пользователь говорит "сейчас занимаемся X", помечай X как `Primary focus` в `handoff/current.md`, но не скрывай остальные активные контексты.
- После значимого этапа обновляй файл задачи и handoff.
- При входном ритуале и перед `commit/push` проверяй `tasks/active/` и `handoff/active/`: завершенные задачи не должны оставаться активными.
- Если задача завершена, перенеси `tasks/active/<task-id>.md` в `tasks/archive/<task-id>.md`, связанный handoff в `handoff/archive/<task-id>.md`, обнови ссылки и убери закрытую задачу из `handoff/current.md`.
- Для работы с несколькими репозиториями одновременно обновляй все затронутые дочерние репозитории и личный штаб: локальный task/handoff/manifest в дочернем проекте, связанный task/handoff штаба и общий `handoff/current.md`.

## Дочерние проекты личного штаба

Если из личного штаба создается отдельный рабочий репозиторий, в его `AGENTS.md` нужно сразу добавить связь с parent repo:

- путь к parent repo;
- task-id и путь к активной задаче;
- путь к parent `handoff/current.md`;
- путь к связанному handoff в `handoff/active/`;
- правило, когда дочерний агент обязан читать и обновлять parent repo.

Дочерний агент должен обновлять parent repo после значимого этапа: смена цели, следующий шаг, риск, статус, решение, созданный manifest/prompt/workflow/research-документ, результат генерации, GPU-run, ревью, блокировка или завершение задачи. Минимум обновления: связанный handoff; если меняется текущий статус или следующий шаг - также запись этого контекста в `handoff/current.md`; если меняется план активной задачи - также файл в `tasks/active/`.

Личный штаб хранит решения, статусы, ссылки и handoff. Тяжелые медиа, модели, рендеры, секреты и реальные `.env` остаются только в дочернем проекте или вне Git согласно его правилам.

## Git-протокол непрерывности

- Перед началом работы проверь статус и подтяни изменения: `git status`, затем `git pull`, если есть удаленный репозиторий.
- Не перезаписывай чужие изменения. Если есть конфликт или незнакомые правки, сначала разберись.
- После значимого этапа предложи или выполни по запросу: обновить документы, сделать `commit` и `push`.
- Для личного штаба предпочтительный режим после значимого `commit` - сразу делать `push`, чтобы GitHub оставался источником правды для другого устройства или чата.
- Коммиты должны быть небольшими и понятными.

## Исследования

Для динамических тем фиксируй:

- источник;
- дату и время проверки в часовом поясе `Europe/Minsk`;
- метод проверки: браузер, официальный сайт, API, поиск, скриншот;
- статус актуальности;
- риск устаревания;
- краткий вывод.

## Оркестрирование

Используй роли как рабочие режимы, а не как бюрократию:

- `Coordinator` - держит цель, план, handoff и качество;
- `Researcher` - проверяет актуальные источники;
- `Builder` - реализует;
- `Verifier` - проверяет факты, тесты, ссылки, скриншоты;
- `Archivist` - чистит мусор, обновляет документы и архив.

Параллельность включай, когда есть независимые части: исследование, реализация, проверка, документация. Для мелких задач не усложняй.

## Безопасность и чистота

- Не храни в репозитории пароли, токены, cookie, приватные ключи, банковские данные и личные документы.
- Не создавай лишние файлы. Временное храни только в явно временных папках, исключенных из Git.
- Перед удалением важных файлов, массовыми изменениями, установкой зависимостей или публикацией спрашивай подтверждение.
