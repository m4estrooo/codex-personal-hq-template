# Codex Personal HQ Template

Публичный шаблон личного штаба для работы с Codex: задачи, проекты, исследования, правила, handoff между чатами и устройствами, GitHub-синхронизация и базовая дисциплина безопасности.

Шаблон рассчитан на персональное использование. Владелец создает из него свой репозиторий, заполняет личные правила и проекты, а затем использует GitHub как источник правды для Codex и для себя.

## Что это дает

- Единое место для задач, проектов, исследований и решений.
- Передача контекста между чатами Codex без зависимости от памяти старого чата.
- Работа с разных устройств через GitHub.
- Понятный жизненный цикл задач: `inbox -> active -> archive`.
- Отдельные handoff-файлы для долгих задач.
- Правила для динамических фактов: цены, объявления, версии сервисов, маркетплейсы, новости.
- Разделение личного штаба и отдельных кодовых проектов.
- Защита от случайного хранения секретов в репозитории.

## Для кого

Шаблон полезен, если вы:

- часто работаете с Codex над разными задачами;
- хотите, чтобы новый чат быстро понимал текущий контекст;
- ведете несколько проектов одновременно;
- используете GitHub как синхронизацию между компьютерами;
- хотите хранить правила, решения и исследования в Markdown, а не только в чатах.

## Базовая архитектура

Репозиторий устроен как личный штаб.

- `AGENTS.md` - проектные правила Codex для этого репозитория.
- `README.md` - инструкция для человека.
- `docs/` - правила, Git-процесс, безопасность, исследования, оркестрирование.
- `tasks/inbox.md` - входящие идеи и неактивные задачи.
- `tasks/active/` - задачи, которые реально сейчас в работе.
- `tasks/archive/` - завершенные задачи.
- `handoff/current.md` - рабочая доска активных контекстов.
- `handoff/active/` - передача контекста по активным задачам.
- `handoff/archive/` - handoff завершенных задач.
- `projects/` - карты проектов, бизнес-направлений и продуктов.
- `research/` - исследования с источниками и датой проверки.
- `templates/` - шаблоны task, handoff, research note, project repo и глобального `AGENTS.md`.
- `automation/` - место под будущие проверки и bootstrap-скрипты.

Главная идея: Codex не должен восстанавливать контекст из памяти. Он читает `handoff/current.md`, открывает нужную active-задачу и связанный handoff, затем продолжает работу.

## Используемые технологии и подходы

- GitHub repository template - создание новых репозиториев из готовой структуры.
- Git - история изменений, commit, push, pull, работа с разных устройств.
- Markdown - человекочитаемые задачи, handoff, исследования и решения.
- `AGENTS.md` - правила поведения Codex внутри репозитория.
- `rg`/ripgrep - быстрый поиск по файлам.
- `.gitignore` - исключение секретов, временных файлов, локального состояния и сборочных артефактов.
- `.env.example` - пример переменных окружения без реальных секретов.
- Handoff-процесс - явная передача цели, статуса, следующего шага, рисков и файлов.
- Research-процесс - фиксация источника, даты, метода проверки и риска устаревания.

## Как создать свой штаб на GitHub

1. Открой template-репозиторий на GitHub.
2. Нажми `Use this template`.
3. Выбери `Create a new repository`.
4. Укажи имя, например `personal-hq`, `codex-hq` или свое название.
5. Выбери видимость:
   - `Private` - рекомендовано для личных задач, проектов и handoff.
   - `Public` - только если внутри не будет личных данных, приватных проектов и рабочих материалов.
6. Создай репозиторий.
7. Склонируй его на компьютер:

```bash
git clone https://github.com/<owner>/<repo>.git
cd <repo>
```

8. Проверь состояние:

```bash
git status
```

## Первичная настройка после создания

1. Открой `templates/codex-global/AGENTS.md`.
2. Замени общие правила на свои личные правила Codex.
3. При необходимости скопируй их в глобальный файл Codex:

```bash
mkdir -p ~/.codex
cp templates/codex-global/AGENTS.md ~/.codex/AGENTS.md
```

4. Открой `AGENTS.md` и уточни правила именно этого репозитория.
5. Открой `tasks/active/initial-setup.md` и выполни чеклист.
6. Открой `handoff/current.md` и проверь текущий фокус.
7. Добавь первые реальные идеи в `tasks/inbox.md`.
8. Добавь первые проекты в `projects/`, если они уже есть.
9. Проверь, что нет секретов:

```bash
git status --short --branch
rg -n --hidden --glob '!.git/**' --glob '!outputs/**' --glob '!work/**' \
  '(AKIA[0-9A-Z]{16}|gh[pousr]_[0-9A-Za-z_]{36,255}|sk-[0-9A-Za-z]{20,}|-----BEGIN (RSA |OPENSSH |EC |DSA )?PRIVATE KEY-----)' .
```

10. Сделай первый commit:

```bash
git add -A
git commit -m "Initialize personal HQ"
git push origin main
```

## Ежедневный процесс

Перед началом работы:

```bash
git status
git pull
```

Дальше:

1. Прочитай `handoff/current.md`.
2. Открой активную задачу из `tasks/active/`.
3. Открой связанный handoff из `handoff/active/`.
4. Создай видимый план Codex.
5. После значимого этапа обнови task-файл, handoff и `handoff/current.md`.
6. Сделай commit и push.

## Жизненный цикл задач

Новая идея или сырая задача:

```text
tasks/inbox.md
```

Задача выбрана в работу:

```text
tasks/active/<task-id>.md
handoff/active/<task-id>.md
```

Задача завершена:

```text
tasks/archive/<task-id>.md
handoff/archive/<task-id>.md
```

Правило: задача не должна одновременно висеть в `inbox` и `active`.

## Как продолжить с другого устройства или нового чата

1. Склонируй репозиторий или открой уже существующую локальную копию.
2. Выполни:

```bash
git pull
```

3. Открой `handoff/current.md`.
4. Открой нужные файлы из `tasks/active/` и `handoff/active/`.
5. Продолжай работу с текущего статуса и следующего шага.

Новый чат должен понимать задачу из файлов, без доступа к старому диалогу.

## Когда создавать отдельный проектный репозиторий

Оставляй проект внутри `projects/`, если это:

- идея;
- исследование;
- бизнес-направление;
- описание продукта;
- контент;
- решения и ссылки.

Создавай отдельный репозиторий, если появляются:

- исходный код;
- зависимости;
- тесты;
- сборка;
- деплой;
- отдельные права доступа;
- отдельная команда.

Для нового кодового проекта скопируй шаблон:

```bash
mkdir my-project
cp -R templates/project-repo/. my-project/
cd my-project
git init -b main
git add -A
git commit -m "Initialize project"
```

Подробности: `docs/project-repos.md`.

## Правила исследований

Для динамических данных нельзя полагаться только на память модели. К динамическим данным относятся:

- цены;
- объявления;
- наличие товаров;
- тарифы;
- версии библиотек и сервисов;
- правила платформ;
- новости;
- характеристики актуальных моделей и продуктов.

Фиксируй:

- источник;
- дату и время проверки;
- часовой пояс;
- метод проверки;
- статус актуальности;
- риск устаревания;
- краткий вывод.

Шаблон: `templates/research-note.md`.

## Безопасность

Не храни в репозитории:

- пароли;
- API-токены;
- cookie;
- приватные ключи;
- банковские данные;
- личные документы;
- реальные `.env` файлы;
- приватные материалы, которые нельзя публиковать.

Можно хранить:

- `.env.example`;
- инструкции;
- публичные ссылки;
- краткие выводы исследований;
- task/handoff без секретов.

Перед публичным push проверь:

```bash
git status
git diff --cached
```

Дополнительно см. `docs/security.md`.

## Рекомендуемые команды

Проверить состояние:

```bash
git status --short --branch
```

Найти файлы:

```bash
rg --files
```

Найти текст:

```bash
rg "поисковая строка"
```

Сохранить этап:

```bash
git add -A
git commit -m "Update task context"
git push
```

## Как объяснить Codex, что делать в новом чате

Минимальная команда:

```text
Прочитай AGENTS.md, затем handoff/current.md, открой текущую active-задачу и связанный handoff. После этого продолжи работу по следующему шагу.
```

Для новой идеи:

```text
Зафиксируй эту идею в tasks/inbox.md, затем предложи следующий конкретный шаг.
```

Для долгой задачи:

```text
Создай active task и handoff, обнови handoff/current.md и веди видимый план.
```

## Что делать перед публикацией собственного штаба

Если вы решили сделать свой штаб публичным:

1. Удалите личные проекты.
2. Удалите приватные handoff-файлы.
3. Удалите реальные задачи с личной информацией.
4. Проверьте `.env`, токены, cookie и ключи.
5. Проверьте историю Git, если секреты когда-либо уже попадали в commit.
6. Оставьте только нейтральные шаблоны и инструкции.

Для личного использования безопаснее держать репозиторий приватным.
