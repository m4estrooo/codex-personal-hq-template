# Общие правила работы

## Язык

- По умолчанию все ответы и документы ведутся на русском.
- Английские термины допустимы, если они общеприняты, но при первом использовании нужно дать русский смысл.

## Факты

- Не использовать память модели как источник актуальных фактов.
- Для цен, объявлений, тарифов, моделей, видеокарт, сервисов, правил площадок и новостей нужна проверка.
- Если проверка невозможна, это нужно явно написать.

## Файлы

- Не создавать файлы без понятного назначения.
- Не плодить черновики в корне проекта.
- Временные материалы хранить в папках, исключенных из Git.
- Любой новый зародыш идеи, задачи или проекта сразу записывать в `tasks/inbox.md`, даже если описание пока неполное.
- Жизненный цикл задачи: `tasks/inbox.md` -> `tasks/active/<task-id>.md` -> `tasks/archive/<task-id>.md`.
- `tasks/inbox.md` хранит все входящие и неактивные задачи: сырые идеи, понятные будущие задачи и задачи в ожидании решения.
- `tasks/backlog.md` не является обязательным этапом и не используется для обычного процесса.
- При переводе задачи в active удалять ее из inbox, чтобы не было двойного статуса.
- Если пользователь явно выбирает задачу как текущую/основную, ее можно перевести из inbox сразу в active с записью причины в task/handoff.
- `handoff/current.md` ведется как рабочая доска активных контекстов. Он может содержать несколько параллельных задач и дочерних репозиториев, один из них помечается как `Primary focus`.
- При параллельной работе над несколькими репозиториями обновлять нужно каждый затронутый репозиторий и личный штаб: локальные task/handoff/manifest, связанный handoff штаба и общий `handoff/current.md`.
- При начале работы и перед `commit/push` проверять `tasks/active/` и `handoff/active/`.
- Завершенные задачи переносить из `tasks/active/` в `tasks/archive/`.
- Связанные handoff-файлы завершенных задач переносить из `handoff/active/` в `handoff/archive/`.
- Если архивируемая задача была текущей, убирать ее из `handoff/current.md` и при необходимости назначать новый `Primary focus`.

## Безопасность

- Не хранить секреты в репозитории.
- Не публиковать приватную информацию без явного решения владельца репозитория.
- При сомнительных действиях выбирать безопасный вариант и фиксировать риск.
